## 하기 싫은데 매일 해야 되는 일

FC Online의 출석/감독모드/웹이벤트 같은 일일 보상은 **하루만 놓쳐도 아깝다**. 그렇다고 매일 데탑 켜서 로그인하고 감독모드 20판 돌리고 4시간 뒤 이벤트 페이지 들어가서 버튼 누르는 건 너무 귀찮다.

그래서 이걸 다 자동화했다. 지금은 **내가 자는 동안 알아서 굴러가고**, 폰으로는 텔레그램 알림만 보면 된다:

```
07:30  ⏰ 오전 7시 - FC Online 자동 시작
07:30  🔌 PC 절전 중. WoL 전송...
07:30  ✅ PC 켜짐! (15초 소요)
07:31  ✅ 2차 로그인 키패드 인식 성공
07:32  ✅ 게임 입장 완료
07:33  ✅ 감독모드 진입 (20판 시작)
11:40  ✅ 감독모드 완료, 게임 종료
11:40  🎮 웹 이벤트 수령 시작...
11:41  ✅ Cowork이 오늘 피파 참여형 웹 이벤트 수령 완료!
11:41  💤 데탑 절전
```

이 글은 이걸 어떻게 쌓아 올렸는지에 대한 기록이다.

## 전체 구조 (3-tier)

```
┌──────────────────────────────────────────────┐
│  [1] cron @ Raspberry Pi 5                   │
│      매일 07:30 auto_start_fconline.py       │
└────────────────┬─────────────────────────────┘
                 │ Wake-on-LAN (매직 패킷)
                 │ + SSH 폴링으로 부팅 대기
                 ▼
┌──────────────────────────────────────────────┐
│  [2] GUI 세션 리스너 @ Windows Desktop       │
│      listener.py (TCP :9999) 상시 대기       │
│      명령 수신 → 자동화 스크립트 실행            │
└────────────────┬─────────────────────────────┘
                 │ subprocess → GUI 세션에서 실행
                 ▼
┌──────────────────────────────────────────────┐
│  [3] start_fconline.py                       │
│      Win32 SendInput + claude -p vision      │
│      Part A/B/C 진행, 각 단계 텔레그램 알림      │
└──────────────────────────────────────────────┘
```

각 티어가 해결하는 문제가 서로 다르다. 그냥 SSH로 스크립트 하나 돌리면 될 것 같지만 **그렇게 못 한다.** 이유는 뒤에서 설명.

## 첫 번째 벽: "SSH에서 pyautogui가 안 먹힌다"

처음에는 단순하게 생각했다. 파이가 크론으로 데탑에 SSH 붙어서 파이썬 스크립트 하나 실행하면 끝 아닌가?

```
sshpass -p <pw> ssh PullyCom@192.168.0.10 \
  'python C:\projects\fconline_auto_event\start_fconline.py'
```

그런데 이게 안 된다. **마우스 클릭이 Chrome에 도달하지 않는다.** 게임 창이 떠 있어도 키 입력이 안 먹힌다.

이유를 파고 들어가 보니:
- `pyautogui`, `Win32 SendInput` 같은 입력 시뮬레이션 API는 **현재 활성화된 GUI 세션(로그인된 바탕화면)의 input queue에 이벤트를 주입**한다
- SSH로 붙은 세션은 "session 0" 계열의 non-interactive 세션이라 GUI와 분리돼 있다
- 그래서 SSH에서 직접 `python start_fconline.py` 하면 SendInput 이벤트가 **어디에도 도달하지 못한다**

### 해결: GUI 세션 안에서 상시 대기하는 리스너

Windows 로그인 직후 자동으로 **TCP 리스너**(`listener.py`)를 GUI 세션 안에서 띄워 둔다. 외부에서 **실행 신호**만 TCP로 보내면, 리스너가 자기 세션 컨텍스트에서 `subprocess.Popen`으로 자동화 스크립트를 실행한다 → SendInput이 정상 작동.

리스너가 받는 명령어는 이런 것들:

```
START_FCONLINE       풀 체인 (게임 실행 → 감독모드 → 4h → 웹이벤트 → 절전)
START_FCONLINE_TEST  테스트 버전
START_GAMDOK         감독모드만
START_EVENTTAKER     웹 이벤트 수령만
SCREENSHOT           현재 화면 캡처
STATUS               미션 진행 여부 조회
```

파이가 외부에서 보낼 때는 `trigger.py`라는 작은 헬퍼로 보낸다:

```python
# trigger.py (요약)
s = socket.create_connection(('localhost', 9999), timeout=5)
s.sendall((sys.argv[1] + '\n').encode())
print(s.recv(1024).decode())
```

응답 코드는 정해뒀다: `OK`, `ALREADY_RUNNING`, `LISTENER_NOT_RUNNING`. 파이 쪽에서 이 코드 보고 텔레그램으로 뭐라고 띄울지 결정한다.

## 두 번째 벽: "2차 로그인 키패드가 매번 배치가 바뀐다"

FC Online(정확히는 Nexon) 로그인에는 **마우스 전용 2차 비밀번호**가 있다. 키보드 입력은 안 받고, 화면에 뜨는 숫자 키패드를 클릭해야 한다. 그런데 이 키패드는 **매번 숫자 배치가 랜덤**이다. 좌표만 하드코딩하면 못 푼다.

이건 사실 **전형적인 비전 문제**다. 스크린샷 찍고 "0부터 9까지 어디에 있는지 알려줘" 하면 된다. OCR이나 템플릿 매칭도 고민했지만,

- Tesseract는 작은 숫자 + 안티앨리어싱에 약함
- 템플릿 매칭은 Nexon이 폰트/배경 바꾸면 다시 튜닝해야 함

### 해결: `claude -p`를 비전 모델처럼 쓰기

Claude Code의 헤드리스 모드(`claude -p`)는 이미지 입력을 받아 사고할 수 있다. Claude Max 구독에 포함되니 **API 추가 비용 없이** 쓸 수 있는 게 컸다.

흐름은 이렇다:

1. 키패드 영역(920,510 ~ 1300,660)만 크롭해서 `numpad.png`로 저장
2. `claude -p --model claude-sonnet-4` 에 "이 이미지에서 숫자 0~9가 어떤 순서로 배치돼 있는지 좌→우, 위→아래 순으로만 알려줘" 프롬프트 + 이미지 전달
3. 응답을 파싱해 `{'0': (x, y), '1': (x, y), ...}` 매핑 생성
4. 좌표는 **고정값**으로 미리 계산해둠 — `x=[960, 1036, 1112, 1188, 1264]`, `y=[550, 625]` (2행 5열)
5. 2차 비밀번호 각 자리를 1.5초 간격으로 클릭

### 같은 기법을 성공 판별에도 씀

Part A(피파 실행) / Part C(감독모드 진입)가 끝나고 나서, **기대하는 화면이 맞는지** 판단해야 한다. 이것도 `claude -p`에 현재 스크린샷 + 레퍼런스 이미지(`ss_reference/part_a_success.png`)를 같이 주고 "이 두 이미지가 동일한 상태를 보여주냐"고 물어본다. 응답이 yes면 다음 단계, no면 재시도.

### 계정 잠금 방어 장치

FC Online은 2차 비밀번호 **5회 틀리면 계정 잠김**. 자동화가 이상하게 돌다가 이런 사고가 나면 내 계정이 날아간다. 그래서 **오류 표시가 화면에 3회 이상 감지되면 즉시 스크립트 중단**하도록 했다. 이것도 스크린샷 → `claude -p`에 "오류 횟수 써 있냐" 질의 조합.

> 이 장치는 실제로 두 번 구해줬다. 한 번은 Chrome이 자동 업데이트 직후 레이아웃이 살짝 밀려서 첫 비번을 잘못 입력했고, 자동 중단 덕에 세 번 만에 멈췄다.

## 세 번째 벽: "PC가 꺼져 있으면 아무것도 안 된다"

데탑을 24시간 켜두기는 전력 낭비다. 평소엔 절전/종료 상태로 두고, 필요할 때 깨워야 한다.

### 파이 → 데탑 Wake-on-LAN

파이 크론이 07:30에 먼저 하는 일:

```python
# auto_start_fconline.py (요약)
if not is_on():
    notify('🔌 PC 절전 중. WoL 전송...')
    subprocess.run(['wakeonlan', DESKTOP_MAC])

    # 최대 120초간 5초 간격으로 부팅 대기
    for i in range(24):
        time.sleep(5)
        if is_on():
            notify(f'✅ PC 켜짐! ({(i+1)*5}초 소요)')
            time.sleep(10)   # 리스너 뜰 시간
            break
    else:
        notify('❌ PC가 120초 안에 켜지지 않음.')
        return
```

`is_on()`은 단순히 `ssh -o ConnectTimeout=5 ... 'echo ok'`가 성공하는지로 판단한다. 소요 시간은 보통 15초 안팎. 부팅 완료 후 리스너가 자동 시작될 때까지 추가로 10초 여유.

### 리스너 자동 시작 등록

데탑 로그인 직후 리스너가 GUI 세션 안에서 실행되도록 **시작프로그램**(shell:startup)에 등록해뒀다. 자동 로그인 설정과 조합해서, WOL 이후 **사용자가 개입 안 해도 리스너가 준비되는** 상태를 만든다.

`START_FCONLINE` 신호가 리스너에 닿으면 이런 순서로 간다.

| Step | 동작 |
| --- | --- |
| 1 | `Win+S` → "chrome" 타이핑 → Enter |
| 2 | "fc" 타이핑 → Enter (자동완성으로 fconline.nexon.com) |
| 3 | 게임시작 버튼 클릭 (1800, 135) |

| 6 | `Alt+Enter` → `Win+↑` (전체화면 → 창모드 최대화) |

마지막에 현재 스크린샷과 `ss_reference/part_a_success.png` 비교로 성공 판별. 실패면 Step 1부터 최대 3회 재시도.

### Part B: 2차 로그인 (위에서 설명한 비전 방식)

| Step | 동작 |
| --- | --- |
| 7 | 키패드 영역 크롭 → `claude -p`로 숫자 배치 인식 |
| 8 | 2차 비밀번호 입력 (자리당 1.5초 간격) |

단순 좌표 클릭 4번. 끝나고 `part_c_success.png`와 비교.

### Step 14: 감독모드 경기 시작

마지막 한 번 클릭으로 20판 시작 버튼 누르면 이 스크립트의 역할은 끝. 이후는 **게임 AI가 알아서 20판 돌리는 동안 기다리는 watcher**가 인계받는다.

## 감독모드가 끝난 뒤: gamdokmode_watcher

풀 체인의 후반부다. `start_fconline.py`가 **"감독모드 시작 버튼을 누른 후 자기 역할 끝"** 이라면, `gamdokmode_watcher.py`는 **"감독모드가 진행되는 4시간을 기다렸다가 뒷정리"** 담당.

순서:

1. 경기 시작 시각을 기록해두고 **4시간 대기**
2. 스크린샷 찍어서 텔레그램 전송 (경기 진짜 끝났는지 내 눈으로 확인용)
3. 게임 프로세스 강제 종료 (`taskkill /F /IM fczf.exe` 등)
4. 감독모드 완료 로그 파일 생성 (공유 폴더에 드랍)
5. 리스너에 `START_EVENTTAKER` 트리거
6. eventTaker 스크립트가 Chrome 띄우고 **참여형 웹이벤트 페이지에서 수령 버튼 클릭**
7. 수령 완료 로그 폴링 (최대 3시간)
8. 성공/실패 텔레그램 보고
9. 데탑 절전 (`rundll32.exe powrprof.dll,SetSuspendState 0,1,0`)

**핵심 아이디어**: watcher도 감독모드 하듯이 계속 화면을 감시하는 게 아니라, **시각 기반 wait** + **파일 기반 시그널**로 분리. CPU/리소스 거의 안 먹음.

## 텔레그램 통합

3층 구조 어디에서나 텔레그램으로 바로 쏠 수 있다:
- **파이**: `requests.post` 로 Telegram Bot API 직접 호출 (sendMessage / sendPhoto)
- **데탑**: 동일 방식 + `PIL.ImageGrab`로 스크린샷 뜨고 sendPhoto
- **별도 봇 명령**: `/start_fconline`, `/start_gamdok`, `/screenshot`, `/wol`, `/sleep` 등을 수동으로도 쓸 수 있음 (라즈베리파이에 돌아가는 Telegram Claude Bot — [별도 글](/projects/telegram-claude-bot) 참고)

모든 알림이 한 채팅방에 몰리니까 **그날의 자동화 흐름이 한눈에 보인다**. 실패 포인트가 생기면 로그 뒤질 필요 없이 텔레그램 타임라인만 스크롤하면 어디서 틀어졌는지 보인다.

## 하면서 배운 것들

- **GUI 세션 분리는 반드시 걸림돌로 튀어나온다**. Windows에서 원격으로 GUI 자동화 돌리려면 "GUI 안에 상시 리스너" 패턴이 정답
- **비전을 `claude -p`로 처리하는 건 꽤 쓸만하다**. API 비용 없이, OCR 튜닝 없이, 프롬프트만 잘 쓰면 변하는 UI도 따라감. 단 **응답 파싱은 방어적으로** — 포맷 깨지는 경우 대비해 JSON으로 답하게 시키는 게 낫다
- **계정 보호 로직은 "성공시킬 방법"이 아니라 "멈출 조건"**. 자동화가 폭주해서 계정을 날리는 것보다 하루 걸러 뛰지 않는 게 훨씬 낫다
- **cron은 봇을 안 거친다.** 텔레그램 봇이 죽어도 cron은 직접 파이썬 스크립트를 부르므로 스케줄이 끊기지 않음. 관심사 분리.
- **WOL + SSH 폴링**만으로 부팅 판정 충분. ping은 방화벽 때문에 의외로 불안정
- **레퍼런스 이미지 기반 성공 판별**이 좌표 기반보다 훨씬 안정적. UI 조금 바뀌어도 비전 모델이 동일 상태로 판단해주면 OK
